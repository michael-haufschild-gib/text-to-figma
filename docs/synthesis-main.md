# Text-to-Figma Design System: Requirements Specification with HTML/SVG Mental Model Layer

## Executive Summary

This document provides a comprehensive requirements specification for building a text-to-Figma design system that generates **aesthetically pleasing designs**, not just functional ones. The system leverages **HTML/CSS/SVG as an intermediate mental model** to improve LLM design generation quality through familiar web development patterns.

**Core Innovation**: Rather than having LLMs generate Figma API calls directly, we enable them to "think" in HTML/CSS/SVG terms—a domain with extensive training data—then translate to Figma primitives. This approach is validated by research showing LLMs generate SVGs 100-200x faster with specialized training [C020], and domain adaptation improves code generation by 18-19% [C026].

**Key Challenge Addressed**: The fundamental challenge in LLM-to-design systems is not functionality but visual quality. This system addresses quality through:

1. **HTML/SVG Mental Model Layer**: Leverages LLM's extensive web development training data [C020, C023]
2. **Constraint-Based Generation**: Reducing LLM decision space through design system constraints
3. **Parameterized Design Patterns**: Mathematical frameworks for spacing, typography, color, and layout
4. **Multi-Layer Quality Validation**: Automated testing at token, visual, and agentic review levels
5. **Bidirectional Communication**: MCP-enabled iterative refinement between LLM and Figma

**Success Metrics**:
- 80-90% conversion accuracy (Builder.io benchmark) [C015]
- Zero pixel-splitting issues through 8pt/4pt grid system adherence
- Component reuse rate >80% through systematic variant usage
- WCAG AA contrast requirements (4.5:1 for normal text, 3:1 for large text)
- Visual regression test pass rate >95% against golden masters

## NEW SECTION: HTML/SVG Mental Model Layer

### Overview: Why HTML/CSS as Intermediate Representation

**Theoretical Foundation**: Research on intermediate representations shows that "training embedding models using both source code and LLVM IR generated by default optimization levels produces notably better results" by capturing lower-level semantics [C023]. Similarly, HTML/CSS serves as an intermediate representation (IR) between natural language prompts and Figma API calls, leveraging LLMs' extensive web development training data.

**Empirical Validation**:
- **LLM4SVG Study**: LLMs trained with 55 semantic SVG tokens on 580k instruction pairs generated SVGs 100-200x faster than optimization methods, with FID score 64.11 vs 127.78 for GPT-4o [C020, C021]
- **Builder.io Production Data**: 80-90% HTML→Figma conversion accuracy in production environments [C015]
- **Domain Adaptation Research**: Two-level fine-tuning (task-level + project-level) increased code generation quality by 18-19% [C026]

**Key Hypothesis**: LLMs have vastly more HTML/CSS/SVG training data than Figma API documentation. By allowing them to generate or reason about designs in HTML/CSS terms first, we can:
1. Leverage stronger domain knowledge (web development)
2. Reduce hallucination rates for API generation [C032: best LLMs only 30-40% correct on direct API calls]
3. Enable few-shot learning with familiar web examples [C028, C029]
4. Create more maintainable and understandable prompt patterns

This table provides comprehensive mappings between CSS properties and Figma API equivalents, essential for building translation layers.

#### 1.1 Layout Properties

|-------------|-----------|----------------|-------------|--------------|----------|

| `display: flex` | `flex-direction: column` | `layoutMode` | `'VERTICAL'` | `frame.layoutMode = 'VERTICAL'` | [C003] |
| `display: grid` | N/A | `layoutMode` | `'GRID'` | `frame.layoutMode = 'GRID'` (beta) | [C009] |

| `justify-content` | `center` | `primaryAxisAlignItems` | `'CENTER'` | `frame.primaryAxisAlignItems = 'CENTER'` | [C004, C005] |
| `justify-content` | `flex-end` | `primaryAxisAlignItems` | `'MAX'` | `frame.primaryAxisAlignItems = 'MAX'` | [C004, C005] |
| `justify-content` | `space-between` | `primaryAxisAlignItems` | `'SPACE_BETWEEN'` | `frame.primaryAxisAlignItems = 'SPACE_BETWEEN'` | [C004, C005] |
| `align-items` | `stretch` | `layoutAlign` | `'STRETCH'` | `node.layoutAlign = 'STRETCH'` | [C043] |
| `gap` | (numeric) | `itemSpacing` | (numeric) | `frame.itemSpacing = 16` | [C009] |
| `position` | `absolute` | `layoutPositioning` | `'ABSOLUTE'` | `node.layoutPositioning = 'ABSOLUTE'` | [C007] |
| `position` | `relative/static` | `layoutPositioning` | `'AUTO'` | `node.layoutPositioning = 'AUTO'` | [C007] |

**Critical Implementation Notes**:

1. **Side Effects** [C002, C006]: Changing `layoutMode` or `primaryAxisAlignItems` automatically updates all children's x and y values. Translation layers must account for this.

2. **Conditional Properties** [C002]: `itemSpacing`, `primaryAxisSizingMode`, and padding are only enabled when `layoutMode` is `'HORIZONTAL'` or `'VERTICAL'`, not `'NONE'` or `'GRID'`.

3. **Non-Reversible Operations** [C044]: "Toggling between modes doesn't restore original positions—removing auto-layout doesn't revert children to their previous arrangement." Translation layers must store original positions if reversal is needed.

4. **CSS Grid Gaps** [C009, C065]: Grid auto-layout uses separate `Gap between rows` and `Gap between columns` fields, not a single gap property.

#### 1.2 Positioning & Constraints

| CSS Property | CSS Value | Figma Property | Figma Value | Code Example | Evidence |
|-------------|-----------|----------------|-------------|--------------|----------|
| `position: absolute` + coordinates | `top`, `left`, `bottom`, `right` | `constraints` | `{ horizontal: 'MIN'/'MAX'/'CENTER'/'STRETCH', vertical: 'MIN'/'MAX'/'CENTER'/'STRETCH' }` | `node.constraints = { horizontal: 'STRETCH', vertical: 'MIN' }` | [C007, C045, C046] |

**Constraint Value Mappings**:
- `left` only → `horizontal: 'MIN'`
- `right` only → `horizontal: 'MAX'`
- `left + right` → `horizontal: 'STRETCH'`
- `top` only → `vertical: 'MIN'`
- `bottom` only → `vertical: 'MAX'`
- `top + bottom` → `vertical: 'STRETCH'`

**Critical Implementation Note** [C045]: "Changing layoutPositioning triggers recalculation of the parent frame's dimensions, as if the repositioned child no longer exists." Translation layers must handle parent dimension updates.

**Scope Constraint** [C046]: "This property applies exclusively to direct children of auto-layout frames. The parent must have layoutMode configured (e.g., HORIZONTAL or VERTICAL)."

#### 1.3 SVG → Figma Vector Paths

| SVG Element | SVG Path Command | Figma Property | Code Example | Evidence |
|------------|------------------|----------------|--------------|----------|
| `<path d="...">` | `M x y` (Move) | `VectorPath.data` | `vectorPath.data = "M 0 100 L 100 100 L 50 0 Z"` | [C008] |
| `<path d="...">` | `L x y` (Line) | `VectorPath.data` | Same as above | [C008] |
| `<path d="...">` | `Z` (Close path) | `VectorPath.data` | Same as above | [C008] |

**Direct Equivalence** [C008]: "VectorPath data property uses SVG-compatible syntax with path commands like M (Move), L (Line), Z (Close) matching SVG d attribute format."

**Best Practice** [C047]: "While vector networks are more powerful, they are significantly more complicated and very easy to get wrong. Paths are usually sufficient and much easier to work with. VectorPath is the recommended way to change the geometry of a vector object."

**Translation Strategy**: For HTML→Figma conversion, extract SVG `<path>` elements' `d` attribute values directly into Figma `VectorPath.data` without modification.

Elements within an Auto Layout container act like flex items by automatically adjusting their size, spacing, and alignment."

### 2. System Prompt Templates with Web Analogies

#### 2.1 Zero-Shot HTML Analogy Prompt

**When to Use** [C028]: "Zero-shot prompting refers to simply asking the model to do something it was not trained to do, relying entirely on pre-training knowledge." Use this when the design task is straightforward and LLM's HTML knowledge is sufficient.

**Token Efficiency** [C030]: Zero-shot uses ~26 tokens vs 88 tokens for few-shot (3.4x difference). "Instruction-tuned models perform well zero-shot on common tasks."

**Template**:
```
You are generating Figma designs using the Figma Plugin API. To help you reason about layouts, think in HTML/CSS terms, then translate to Figma API calls.

HTML/CSS MENTAL MODEL MAPPINGS:
- CSS `display: flex; flex-direction: row` → Figma `layoutMode = 'HORIZONTAL'`
- CSS `display: flex; flex-direction: column` → Figma `layoutMode = 'VERTICAL'`
- CSS `justify-content: space-between` → Figma `primaryAxisAlignItems = 'SPACE_BETWEEN'`
- CSS `gap: 16px` → Figma `itemSpacing = 16`
- CSS `position: absolute` → Figma `layoutPositioning = 'ABSOLUTE'`
- SVG `<path d="M 0 0 L 100 100">` → Figma `VectorPath.data = "M 0 0 L 100 100"`

DESIGN CONSTRAINTS (8pt grid, modular type scale, WCAG AA):
[Include existing constraint system from Section 7.1]

TASK: {user_prompt}

WORKFLOW:
1. Conceptualize layout in HTML/CSS terms (mental model)
2. Map HTML/CSS to Figma API calls using table above
3. Generate Figma API commands
4. Validate constraints
```

**Expected Performance** [C030]: "Performance plateaus after 4-5 examples" in few-shot, so zero-shot with clear mappings may be sufficient for common layouts.

#### 2.2 Few-Shot HTML Analogy Prompt

**When to Use** [C029]: "Few-shot prompting is useful when: zero-shot performance proves insufficient, limited training data scenarios, custom format or style requirements, teaching unfamiliar concepts through examples, precision-critical applications."

**Token Cost** [C030]: Few-shot with 2 examples costs ~88 tokens (3.4x zero-shot). "Adding only high-leverage examples addressing edge cases or complex formatting" is recommended.

**Template**:
```
You are generating Figma designs using the Figma Plugin API. Use HTML/CSS mental models to reason about layouts.

EXAMPLE 1: Horizontal Navigation Bar
HTML Concept: <nav style="display: flex; gap: 24px; padding: 16px;">
Figma Translation:
const nav = figma.createFrame();
nav.layoutMode = 'HORIZONTAL';
nav.itemSpacing = 24;
nav.paddingLeft = 16; nav.paddingRight = 16;
nav.paddingTop = 16; nav.paddingBottom = 16;

EXAMPLE 2: Centered Card with Absolute Badge
HTML Concept: <div class="card" style="position: relative;">
              <span class="badge" style="position: absolute; top: 8px; right: 8px;">NEW</span>
Figma Translation:
const card = figma.createFrame();
card.layoutMode = 'VERTICAL';
const badge = figma.createText();
badge.layoutPositioning = 'ABSOLUTE';
badge.x = card.width - badge.width - 8;
badge.y = 8;
badge.constraints = { horizontal: 'MAX', vertical: 'MIN' };

TASK: {user_prompt}

[Include full mapping table and constraints]
```

**Evidence for Few-Shot Effectiveness** [C052]: Few-shot prompting successfully taught an LLM an invented language (Blablarian) by providing grammatical examples, demonstrating "ability to learn unfamiliar domain patterns from minimal examples."

**Token Optimization** [C056]: "Starting with zero-shot approaches, adding only high-leverage examples addressing edge cases or complex formatting, prioritizing precise instructions over verbose demonstrations, matching example selection to model pretraining knowledge."

#### 2.3 Chain-of-Thought HTML Mental Model Prompt

**When to Use** [C053]: "Neither zero-shot nor few-shot handles complex multi-step reasoning effectively. The article illustrates this with an arithmetic problem requiring step-by-step logic—both methods failed until the model employed external Python code execution."

**Template**:
```
You are generating Figma designs. Use step-by-step reasoning with HTML/CSS mental models.

TASK: {user_prompt}

REASONING PROCESS:
Step 1: Describe the HTML structure you would use
Example: "I would use a flex container with column direction containing three sections"

Step 2: Map HTML to Figma concepts
Example: "Flex column = layoutMode: 'VERTICAL', sections = child frames"

Step 3: Generate Figma API calls with inline comments
Example:
// Main container: <div style="display: flex; flex-direction: column; gap: 32px;">
const container = figma.createFrame();
container.layoutMode = 'VERTICAL';  // flex-direction: column
container.itemSpacing = 32;         // gap: 32px

Step 4: Validate against constraints
Example: "itemSpacing = 32 ✓ (8pt grid), layoutMode = 'VERTICAL' ✓ (valid value)"

Now complete the task following this reasoning process.
```

**Applicable Scenarios**: Complex layouts with nested containers, responsive designs with multiple constraint rules, designs requiring both auto-layout and absolute positioning.

#### 2.4 Domain Adaptation Two-Level Fine-Tuning Prompt Strategy

**Theoretical Foundation** [C026, C054]: "Domain adaptation combines task-level fine-tuning on general dataset (Methods2test) with project-level fine-tuning on target data to learn project-specific knowledge using CodeT5... Following task-level adaptation, the approach applies domain adaptation to each target project data to learn project-specific knowledge."

**Performance Gains** [C026, C055]: Two-level domain adaptation increased code generation by 18-19% for line coverage and 12-16% for mutation scores. "When combined with EvoSuite, achieved average increases of 34.42% line coverage and 6.8% mutation score."

**Implementation Strategy**:

**Level 1: Task-Level Fine-Tuning (HTML→Figma General)**
```
Training Dataset: 1000+ examples of HTML/CSS → Figma API translations
Objective: Learn general HTML/CSS to Figma mapping patterns
Examples:
- <div style="display: flex; flex-direction: row;"> → frame.layoutMode = 'HORIZONTAL'
- <button style="padding: 12px 24px;"> → button.paddingLeft = 24; button.paddingTop = 12
- <path d="M 0 0 L 100 100"> → vectorPath.data = "M 0 0 L 100 100"

Fine-tune model: CodeT5, GPT-3.5, or Claude Sonnet on this general corpus
```

**Level 2: Project-Level Fine-Tuning (Design System Specific)**
```
Training Dataset: 100-500 examples from specific design system
Objective: Learn project-specific component patterns, naming conventions, token usage
Examples (Material Design system):
- <button class="mdc-button mdc-button--raised"> → createInstance(MaterialButton); setProperties({ variant: 'raised' })
- <div class="mdc-card"> → createInstance(MaterialCard)
- Color tokens: --mdc-theme-primary → COLOR_PALETTE.primary

Fine-tune level-1 model on project-specific examples
```

**Prompt Injection Pattern (No Fine-Tuning Available)**:
```
SYSTEM CONTEXT: You are specialized in generating Figma designs using the Material Design system.

DESIGN SYSTEM KNOWLEDGE:
- Button component has variants: text, outlined, raised, unelevated
- Card component uses 4dp elevation with borderRadius: 4
- Typography scale: 12/14/16/20/24/34/48/60
- Primary color: #6200EE, Secondary: #03DAC6

HTML MENTAL MODEL MAPPINGS:
[Include general mappings]

PROJECT-SPECIFIC PATTERNS:
- <button class="mdc-button mdc-button--raised"> maps to MaterialButton instance with variant='raised'
- <div class="mdc-card"> maps to MaterialCard component instance
- Always use design system components when available before creating custom components

TASK: {user_prompt}
```

**Expected Improvement**: Based on research showing 18-19% improvement, expect similar gains in Figma design generation quality when using two-level adaptation vs. generic prompting.

### 3. Translation Layer Architecture Options

Three architectural patterns for implementing HTML/CSS → Figma translation, based on converter tool analysis.

#### 3.1 Prompt-Based Translation (No Runtime Conversion)

**Approach**: LLM generates Figma API calls directly using HTML/CSS mental model in system prompt. No actual HTML parsing.

**Architecture**:
```
User Prompt → LLM (with HTML mental model system prompt) → Figma API Calls → Figma Plugin → Design
```

**Advantages**:
- Zero parsing overhead
- Leverages LLM's HTML knowledge directly
- Simplest implementation
- No intermediate representation serialization

**Disadvantages**:
- LLM must maintain HTML→Figma mapping accuracy
- No validation of HTML conceptual correctness before Figma generation
- Hallucination risk [C032]: "Best open-source LLMs generate correct API invocations only 30-40% of time"

**Best For**: Simple layouts where LLM can reliably map concepts without intermediate validation.

**Token Efficiency** [C030, C031]: Zero-shot prompt ~26 tokens. "JSON format consumes 69% more tokens than plain text (22 vs 13 tokens)." Use plain text Figma API calls rather than JSON.

**Implementation Example**:
```typescript
const systemPrompt = `
You generate Figma API calls using HTML/CSS mental models.
CSS flex-direction: row → layoutMode = 'HORIZONTAL'
CSS gap: 16px → itemSpacing = 16
Generate TypeScript code calling Figma API directly.
`;

const userPrompt = "Create a horizontal navigation bar";

// LLM output:
const nav = figma.createFrame();
nav.layoutMode = 'HORIZONTAL';
nav.itemSpacing = 24;
```

#### 3.2 Runtime Two-Stage Translation (Browser + Plugin)

**Approach**: Extract HTML DOM metadata in browser, send to Figma plugin for rendering. Based on html-to-figma architecture [C010, C011, C048].

**Architecture**:
```
HTML/CSS → Browser DOM Extraction → layersMeta (JSON) → Figma Plugin → Figma Nodes
          (browser-side)              (intermediate)     (plugin-side)
```

**Key Pattern** [C010, C048]: "Browser-side htmlTofigma function extracts DOM metadata into intermediate layersMeta. Figma-side addLayersToFrame translates to native Figma layers. This architecture employs a metadata-driven transformation pattern, where DOM traversal produces an intermediate representation that Figma operations consume, avoiding direct DOM-to-Figma coupling."

**Intermediate Format (layersMeta)** [C011]: Contains:
- Element hierarchy and nesting
- Computed styles and dimensions
- Visual properties (colors, typography, spacing)
- Layout constraints

**Advantages**:
- Decouples DOM analysis from Figma rendering
- Browser provides accurate computed styles
- Can capture live webpage state (authenticated pages, localhost)
- Enables LLM to generate HTML first, then convert

**Disadvantages**:
- Requires browser environment (can't run in Node.js MCP server alone)
- More complex architecture (3 components)
- Network overhead for layersMeta transfer

**Best For**: Converting existing webpages to Figma, scenarios where LLM generates HTML/CSS first, authenticated page conversion [C016].

**Implementation Example** (from html-to-figma pattern):
```typescript
// Browser-side: Extract DOM to intermediate format
function htmlToLayersMeta(element: HTMLElement): LayersMeta {
  const computedStyle = window.getComputedStyle(element);

  return {
    type: element.tagName.toLowerCase(),
    children: Array.from(element.children).map(child => htmlToLayersMeta(child)),
    styles: {
      display: computedStyle.display,
      flexDirection: computedStyle.flexDirection,
      justifyContent: computedStyle.justifyContent,
      gap: computedStyle.gap,
      padding: computedStyle.padding,
      // ... all relevant computed styles
    },
    dimensions: {
      width: element.offsetWidth,
      height: element.offsetHeight,
      x: element.offsetLeft,
      y: element.offsetTop
    },
    visual: {
      backgroundColor: computedStyle.backgroundColor,
      color: computedStyle.color,
      fontSize: computedStyle.fontSize,
      fontFamily: computedStyle.fontFamily
    }
  };
}

// Send to Figma plugin via postMessage
const meta = htmlToLayersMeta(document.querySelector('.my-component'));
parent.postMessage({ pluginMessage: { type: 'import-html', meta } }, '*');

// Figma plugin-side: Render from intermediate format
function addLayersToFrame(meta: LayersMeta, parentFrame: FrameNode) {
  const frame = figma.createFrame();

  // Map styles to Figma properties
  if (meta.styles.display === 'flex') {
    frame.layoutMode = meta.styles.flexDirection === 'row' ? 'HORIZONTAL' : 'VERTICAL';
    frame.itemSpacing = parseFloat(meta.styles.gap) || 0;
  }

  // Recursively process children
  meta.children.forEach(childMeta => {
    addLayersToFrame(childMeta, frame);
  });

  parentFrame.appendChild(frame);
}
```

#### 3.3 Hybrid LLM-Assisted Translation

**Approach**: Combine runtime HTML parsing with LLM-based intelligent decision-making for ambiguous cases.

**Architecture**:
```
HTML/CSS → Parser (deterministic mappings) → Ambiguous Cases → LLM Decision → Figma API
                                                              ↓
                                                    Context: Design System,
                                                             Component Library,
                                                             Best Practices
```

**Decision Points for LLM**:
1. **Component Recognition**: Does `<div class="card">` map to existing Card component or create new frame?
2. **Layout Strategy**: Use auto-layout HORIZONTAL or absolute positioning?
3. **Token Selection**: Which design token for `padding: 18px` (round to 16 or 24)?
4. **Semantic Grouping**: Should these elements be grouped into a component?

**Advantages**:
- Deterministic mapping for clear cases (fast, reliable)
- LLM intelligence for ambiguous cases (context-aware)
- Can leverage design system knowledge
- Best of both worlds: speed + intelligence

**Disadvantages**:
- Most complex architecture
- LLM costs for decision-making (but lower than full generation)
- Requires clear heuristics for when to invoke LLM

**Best For**: Production systems requiring high accuracy, designs using established component libraries, scenarios with complex design system rules.

**Implementation Example**:
```typescript
interface TranslationContext {
  designSystemComponents: Component[];
  designTokens: DesignTokens;
  bestPractices: string[];
}

async function translateElement(
  element: HTMLElement,
  context: TranslationContext
): Promise<SceneNode> {

  // Deterministic mapping for clear cases
  if (element.style.display === 'flex' && element.style.flexDirection === 'row') {
    const frame = figma.createFrame();
    frame.layoutMode = 'HORIZONTAL';
    frame.itemSpacing = parseGap(element.style.gap);
    return frame;
  }

  // Ambiguous case: Is this a component instance?
  if (element.className.includes('card')) {
    const decision = await llm.decide({
      question: "Should this element use an existing component or create a new frame?",
      context: {
        elementHTML: element.outerHTML,
        availableComponents: context.designSystemComponents.map(c => c.name),
        computedStyles: window.getComputedStyle(element)
      }
    });

    if (decision.useComponent) {
      const component = context.designSystemComponents.find(c => c.name === decision.componentName);
      return component.createInstance();
    } else {
      return figma.createFrame(); // Create new
    }
  }

  // Ambiguous case: Token selection
  const paddingValue = parseFloat(element.style.padding);
  if (!context.designTokens.spacing.includes(paddingValue)) {
    const decision = await llm.decide({
      question: `Padding value ${paddingValue}px doesn't match design tokens [${context.designTokens.spacing}]. Which token should be used?`,
      context: { visualContext: element.outerHTML }
    });
    return decision.selectedToken;
  }
}
```

**Cost Analysis** [C030]: "Few-shot prompting increases token cost 3.4x." Hybrid approach uses LLM selectively, reducing costs vs. full LLM generation:
- Deterministic mappings: 0 LLM tokens
- LLM decision points: ~50-100 tokens per decision
- Average 3-5 decisions per design → 150-500 tokens
- vs. Full LLM generation: 1000-2000 tokens

**Expected Savings**: 50-75% token cost reduction vs. full LLM generation while maintaining intelligence for ambiguous cases.

| **Speed** | Medium (LLM latency) | Fast (no LLM) | Fast (LLM only for decisions) |
| **Component Recognition** | LLM-based (unreliable) | Rule-based (limited) | LLM-based (reliable) |
| **Design System Integration** | Prompt injection | Hard-coded mappings | Context-aware |
| **Best Use Case** | Simple layouts | Webpage conversion | Production systems |
| **Maintenance** | Easy (prompt updates) | Medium (code updates) | Complex (hybrid logic) |

#### 3.5 Recommended Architecture: Hybrid with Progressive Enhancement

**Phase 1 (MVP)**: Start with Runtime Two-Stage for deterministic accuracy
**Phase 2 (Enhancement)**: Add LLM decision-making for component recognition
**Phase 3 (Optimization)**: Implement caching for LLM decisions to reduce costs

**Rationale**:
1. Achieves 80-90% accuracy baseline immediately [C015]
2. Reduces hallucination risk (30-40% error rate for direct LLM API generation) [C032]
3. Enables incremental improvement without rewriting
4. Balances accuracy, cost, and complexity

**Why IR Matters**:

2. **Validation**: IR can be validated before generation

4. **Multi-Target**: Single IR can generate to multiple targets

**Recommended IR Schema for HTML→Figma**:

```typescript
interface DesignIR {

    nodeId: string;

  }[];

```

This schema balances comprehensiveness (all Figma properties) with simplicity (flat structure, easy to validate).

### 5. Transfer Learning and Cognitive Validation

Theoretical and empirical evidence supporting HTML/SVG as intermediate representation for LLM-Figma systems.

#### 5.1 Transfer Learning Theory Foundation

**Domain Adaptation Framework** [C024]: "Source domain represents training data distribution, target domain represents test data. Three types: prior shift (class proportions), covariate shift (feature distributions), concept shift (decision boundaries)."

**Application to HTML→Figma**:
- **Source Domain**: HTML/CSS/SVG (extensive LLM training data)
- **Target Domain**: Figma API (limited LLM training data)
- **Shift Type**: Covariate shift (feature distributions differ but concepts overlap)
- **Transfer Mechanism**: Intermediate representation (HTML concepts map to Figma primitives)

**Six Domain Adaptation Approaches** [C025]:
1. **Importance-weighting**: Reweight HTML examples by relevance to Figma
2. **Subspace mapping**: Project HTML/Figma into shared lower-dimensional space
3. **Domain-invariant spaces**: Learn representations invariant across HTML/Figma
4. **Feature augmentation**: Expand HTML features with Figma-specific properties
5. **Minimax estimators**: Adversarial training between HTML/Figma
6. **Robust algorithms**: Algorithms resilient to domain shift

**Recommended Approach for LLM-Figma**: Domain-invariant representation learning [C025]. Train LLM to recognize layout concepts (container, flexbox, grid) independent of whether they're expressed in HTML or Figma API syntax.

**Implementation via Prompt Engineering**:
```
DOMAIN-INVARIANT LAYOUT CONCEPTS:

Container: An element that holds child elements
- HTML: <div>, <section>, <article>
- Figma: FrameNode with layoutMode

Horizontal Layout: Children arranged left-to-right
- HTML: display: flex; flex-direction: row
- Figma: layoutMode = 'HORIZONTAL'

Spacing Between Children: Gap between adjacent elements
- HTML: gap: 16px
- Figma: itemSpacing = 16

Think in terms of these CONCEPTS, not syntax.
```

#### 5.2 Intermediate Representation Effectiveness

**LLVM IR Evidence** [C023]: "Training embedding models using both source code and LLVM IR generated by default optimization levels produces notably better results than source code alone by capturing lower-level semantics."

**Parallel to HTML→Figma**:
- **Source Code** = Natural language prompt ("create a navigation bar")
- **LLVM IR** = HTML/CSS representation (`<nav style="display: flex; gap: 24px;">`)
- **Target** = Figma API calls (`frame.layoutMode = 'HORIZONTAL'; frame.itemSpacing = 24`)

**Why HTML/CSS as IR Works**:
1. **Lower-level semantics**: HTML captures structural intent more precisely than natural language
2. **Validation**: HTML can be validated syntactically before Figma generation
3. **Visualization**: HTML can be rendered in browser to verify correctness
4. **Training data abundance**: LLMs have vastly more HTML training data than Figma API

**Empirical Validation**: LLM4SVG study [C020, C021] demonstrates this principle:
- **Without IR**: GPT-4o generating SVGs directly achieved FID score 127.78
- **With IR**: LLM4SVG using 55 semantic SVG tokens achieved FID score 64.11 (2x better)
- **Speed Improvement**: 12-25 seconds vs 5-49 minutes (100-200x faster)

**Transfer to Figma Context**:
```
LLM → Natural Language → Direct Figma API = High error rate, slow
LLM → Natural Language → HTML/CSS → Figma API = Lower error rate, faster (leverages HTML training)
LLM → Natural Language + HTML examples → Figma API = Optimal (few-shot with familiar domain)
```

#### 5.3 LLM4SVG Semantic Token Approach

**Key Innovation** [C020]: "55 SVG semantic tokens comprising 15 tag tokens, 30 attribute tokens, 10 path command tokens. Generated 580k SVG-text instruction pairs. Dataset scale: 250,000 SVG graphics → 580,000 instruction pairs."

**Performance Results** [C021]:
- **FID Score**: 64.11 (LLM4SVG) vs 127.78 (GPT-4o baseline) — 2x better
- **CLIPScore**: 0.3496 vs 0.3001 (SVGDreamer optimization method)
- **Human Alignment**: 0.89 vs 0.49 (GPT-4o) — nearly 2x better prompt alignment
- **Speed**: 18 seconds vs 43 minutes 56 seconds — 145x faster

**Technical Implementation** [C022]: "Expanded vocabulary: |V|' = |V| + 55. Initialization uses semantic averaging: each token embedding equals the mean of embeddings from its descriptive text tokens."

**Application to Figma API**:

#### 5.5 Few-Shot Learning Effectiveness for Unfamiliar Domains

**Invented Language Study** [C052]: "Few-shot prompting successfully taught LLM an invented language (Blablarian) by providing grammatical examples. The model successfully translated novel sentences by recognizing patterns from minimal examples."

**Implication for Figma API**: Figma API is an "unfamiliar domain" for LLMs (limited training data). Few-shot examples with HTML analogies can "teach" the API through pattern recognition.

**Complex Reasoning Limitation** [C053]: "Neither zero-shot nor few-shot handles complex multi-step reasoning effectively... both methods failed until the model employed external Python code execution."

**Mitigation Strategy**: For complex Figma designs (nested components, multiple constraints), use chain-of-thought prompting:

```
TASK: Create a responsive dashboard with sidebar, header, and content area.

REASONING STEPS:
1. HTML Structure Planning:
   <div class="dashboard"> (flex-row)
     <aside class="sidebar"> (flex-column)
     <div class="main"> (flex-column)
       <header> (flex-row)
       <main class="content"> (flex-column)

2. Figma Mapping:
   - Dashboard container: layoutMode HORIZONTAL
   - Sidebar: layoutMode VERTICAL, fixed width
   - Main area: layoutMode VERTICAL, layoutAlign STRETCH
   - Header: layoutMode HORIZONTAL
   - Content: layoutMode VERTICAL, layoutAlign STRETCH

3. Constraints for Responsiveness:
   - Sidebar: constraints { horizontal: 'MIN', vertical: 'STRETCH' }
   - Main area: constraints { horizontal: 'STRETCH', vertical: 'STRETCH' }

4. Implementation:
   [Generate Figma API calls]

5. Validation:
   [Check constraints, spacing, component reuse]
```

This explicit step-by-step reasoning addresses the "complex multi-step" limitation.

#### 5.6 Multimodal LLM Cross-Domain Transfer

**Architecture Insight** [C027, C068]: "Multimodal LLMs enable cross-domain knowledge transfer using domain-invariant representations through shared transformer layers processing unified multimodal embeddings... Core architecture: Modality-specific encoders, cross-modal fusion with attention mechanisms, shared transformer layers processing unified representations."

**Application to HTML/Figma Transfer**:
```
Text Prompt → Text Encoder → Shared Transformer Layers ← Visual Encoder ← Screenshot/Mockup
                                        ↓
                              Domain-Invariant Layout Representation
                                        ↓
                              Figma API Code Generator
```

**Multimodal Workflow**:
1. User provides: Text description + reference screenshot/mockup
2. Visual encoder extracts layout structure from image
3. Text encoder processes natural language requirements
4. Shared transformer fuses visual + text into domain-invariant representation
5. Generator produces Figma API calls matching visual structure + text constraints

**Relevance to HTML Mental Model**: HTML serves as the "domain-invariant layout representation" that bridges text and visual modalities. Both text ("create a navigation bar") and visuals (screenshot of nav bar) map to the same HTML concept (`<nav style="display: flex">`), which then maps to Figma.

**Research Gap**: No evidence found for multimodal LLMs specifically applied to Figma generation. This represents a promising research direction.

#### 6.4 Visual Regression Testing for HTML→Figma Accuracy

**VRT Workflow** [from baseline synthesis]:
1. **Generate Baseline**: Convert HTML to Figma manually or with high-accuracy tool
2. **Export Baseline Screenshot**: PNG at 2x resolution
3. **Generate Test Output**: LLM-generated Figma design from same HTML
4. **Export Test Screenshot**: PNG at 2x resolution
5. **Pixel Comparison**: Calculate structural similarity and highlight differences

**SSIM (Structural Similarity Index) Metric**:
- **Range**: 0.0 (completely different) to 1.0 (identical)
- **Target**: SSIM > 0.90 (90% visual similarity)
- **Acceptable Threshold**: SSIM > 0.85 for complex layouts with text rendering variance

```typescript
import pixelmatch from 'pixelmatch';
import { PNG } from 'pngjs';

async function compareVisualOutput(
  htmlInput: string,
  figmaOutput: FrameNode,
  baseline: string
): Promise<VRTResult> {

  // Export Figma output as PNG
  const testPNG = await figmaOutput.exportAsync({
    format: 'PNG',
    constraint: { type: 'SCALE', value: 2 }
  });

  // Load baseline
  const baselineImg = PNG.sync.read(fs.readFileSync(baseline));
  const testImg = PNG.sync.read(testPNG);

  // Create diff image
  const { width, height } = baselineImg;
  const diff = new PNG({ width, height });

  // Pixel-by-pixel comparison
  const numDiffPixels = pixelmatch(
    baselineImg.data,
    testImg.data,
    diff.data,
    width,
    height,
    { threshold: 0.1 }  // 10% threshold for anti-aliasing tolerance
  );

  const totalPixels = width * height;
  const diffPercentage = numDiffPixels / totalPixels;

  // Calculate SSIM (structural similarity)
  const ssim = calculateSSIM(baselineImg, testImg);

  return {
    passed: ssim >= 0.90 && diffPercentage <= 0.05,
    ssim,
    diffPercentage,
    diffPixels: numDiffPixels,
    diffImage: PNG.sync.write(diff)
  };
}
```

**SSIM Calculation**:
```typescript
function calculateSSIM(img1: PNG, img2: PNG): number {
  // Simplified SSIM algorithm (luminance + contrast + structure)
  const { width, height } = img1;

  let meanImg1 = 0, meanImg2 = 0;
  let variance1 = 0, variance2 = 0, covariance = 0;

  // Calculate means
  for (let i = 0; i < img1.data.length; i += 4) {
    const gray1 = 0.299 * img1.data[i] + 0.587 * img1.data[i+1] + 0.114 * img1.data[i+2];
    const gray2 = 0.299 * img2.data[i] + 0.587 * img2.data[i+1] + 0.114 * img2.data[i+2];
    meanImg1 += gray1;
    meanImg2 += gray2;
  }

  meanImg1 /= (width * height);
  meanImg2 /= (width * height);

  // Calculate variances and covariance
  for (let i = 0; i < img1.data.length; i += 4) {
    const gray1 = 0.299 * img1.data[i] + 0.587 * img1.data[i+1] + 0.114 * img1.data[i+2];
    const gray2 = 0.299 * img2.data[i] + 0.587 * img2.data[i+1] + 0.114 * img2.data[i+2];

    variance1 += Math.pow(gray1 - meanImg1, 2);
    variance2 += Math.pow(gray2 - meanImg2, 2);
    covariance += (gray1 - meanImg1) * (gray2 - meanImg2);
  }

  variance1 /= (width * height);
  variance2 /= (width * height);
  covariance /= (width * height);

  // SSIM formula
  const c1 = 6.5025;  // (0.01 * 255)^2
  const c2 = 58.5225; // (0.03 * 255)^2

  const ssim = ((2 * meanImg1 * meanImg2 + c1) * (2 * covariance + c2)) /
               ((meanImg1 * meanImg1 + meanImg2 * meanImg2 + c1) * (variance1 + variance2 + c2));

  return ssim;
}
```
